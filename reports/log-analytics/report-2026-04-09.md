## Pattern & Probable Cause

The two exception types are tightly coupled:

1. **`TaskCanceledException`** — A request (or background task driven by a request) had its `CancellationToken` canceled. This is typically triggered by:
   - Client disconnecting early (browser navigation, mobile app backgrounding, proxy timeout).
   - A server-side timeout (e.g., `HttpClient` timeout, `IHostedService` shutdown, or explicit `CancellationTokenSource` timeout).

2. **`ObjectDisposedException` on `IFeatureCollection`** — After the `CancellationToken` fires, code continued executing and attempted to access the `HttpContext` (or one of its features), which had already been disposed as part of request teardown.

The counts being similar (186 vs 141) suggests most `TaskCanceledException` events are followed by an `ObjectDisposedException`, with ~45 cases where the disposed-context access did not occur (e.g., the code path exited cleanly after catching the cancellation).

**Most likely scenario:** A downstream call (database, HTTP dependency, message queue) hangs or is slow. The client disconnects. ASP.NET Core signals cancellation and disposes the `HttpContext`. Awaited code resumes after the disposal and touches the disposed context.

---

## Remediation Ideas

1. **Guard against `HttpContext` access after cancellation.**  
   Check `HttpContext.RequestAborted.IsCancellationRequested` before accessing context features in async continuations. Alternatively, capture needed values (headers, route data) early in the pipeline.

2. **Swallow or re-throw `TaskCanceledException` deliberately.**  
   In controllers/middleware that handle client-disconnect scenarios, catch `OperationCanceledException` (parent of `TaskCanceledException`) and return gracefully without further context access.

3. **Review middleware ordering.**  
   Ensure exception-handling middleware (`UseExceptionHandler` / `UseDeveloperExceptionPage`) runs before any middleware that accesses `HttpContext` features, so disposed-context exceptions are caught rather than logged as unhandled.

4. **Investigate the 12:44–15:08 UTC window.**  
   Check deployment events, dependency health (database, downstream APIs), and load metrics in that window to identify what caused the spike. If it coincides with a deploy or a dependency degradation, address that root cause.

5. **Add structured logging for client disconnects.**  
   Log `RequestAborted` cancellations at `Information` level separately so they can be filtered from genuine errors in dashboards.
