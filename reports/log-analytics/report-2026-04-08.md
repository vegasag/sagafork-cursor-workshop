The `HttpContext` / `IFeatureCollection` backing an ASP.NET Core request was already disposed by the time `ForbiddenMiddleware` attempted to access it. This is a known race condition in the Azure Functions **isolated worker + ASP.NET Core integration** when the function host recycles or a request is aborted mid-pipeline.

---

## Probable Causes

1. **Host restart / deployment during the incident window** — the synchronized start time (~12:53) for both exception types strongly suggests a rolling restart or new deployment kicked off at that moment.
2. **Race condition in middleware lifecycle** — `FunctionsHttpProxyingMiddleware` disposes the proxied `HttpContext` before all downstream middleware (e.g. `ForbiddenMiddleware`) finishes reading from it. This is more likely to surface under load or during restart drain.
3. **Graceful-shutdown cancellation** — `TaskCanceledException` is often expected during shutdown; whether each cancellation is truly graceful depends on caller handling.

---

## Remediation Ideas

### Short-term
- **Guard `ForbiddenMiddleware`** against disposed context:
  ```csharp
  public async Task Invoke(FunctionContext context, FunctionExecutionDelegate next)
  {
      try { /* existing logic */ }
      catch (ObjectDisposedException) { /* log and return; request already gone */ }
  }
  ```
- Apply the same defensive catch in `ServiceUnavailableMiddleware` and `CorsMiddleware` — both appear in the stack and could be affected.

### Medium-term
- **Update `Microsoft.Azure.Functions.Worker.Extensions.Http.AspNetCore`** — several patch releases have addressed IFeatureCollection lifetime issues in the isolated worker; check the release notes for fixes.
- **Review middleware ordering** — ensure `FunctionsHttpProxyingMiddleware` is registered last (outermost) so it does not dispose the context before inner middleware completes.
- **Correlate with deployment history** — query `AppMetrics` or deployment logs for 2026-04-07 ~12:53 UTC to confirm a restart/deploy event aligns with exception onset.

### Long-term
- Add a **custom alert rule** in Azure Monitor that fires when `IFeatureCollection has been disposed` count exceeds a threshold per hour, enabling faster on-call response.
- Consider a **canary / blue-green deployment** strategy to reduce in-flight request disruption during restarts.
