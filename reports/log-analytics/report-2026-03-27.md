## Exception 1 — TaskCanceledException (204 occurrences)

**Message:** `A task was canceled.`
**ProblemId:** `System.Threading.Tasks.TaskCanceledException at System.Runtime.CompilerServices.TaskAwaiter.ThrowForNonSuccess:38`

### Probable causes
- HTTP client requests hitting a timeout (`HttpClient.Timeout`) before receiving a response — especially likely when the downstream service is throttling (see Exception 2 below).
- `CancellationToken` passed by the caller (e.g. a browser navigating away, or an ASP.NET request pipeline cancellation) being triggered before the awaited operation completes.
- Background `Task` chains where an outer cancellation propagates inward.

### Remediation ideas
- Correlate with the 429 exceptions — many task cancellations may be direct consequences of slow/retried HTTP calls that eventually time out.
- Review `HttpClient` timeout configuration; consider increasing it or implementing retry-with-backoff (e.g. Polly) so transient 429s are retried before the `CancellationToken` fires.
- Check whether ASP.NET request-abort cancellation is propagating into long-running downstream calls; guard with `CancellationToken.None` or a linked token with a dedicated timeout where appropriate.
- Add structured logging around the cancellation site to capture which endpoint/operation is involved.

## Exception 2 — HttpRequestException HTTP 429 Too Many Requests (125 occurrences)

**Message:** `Response status code does not indicate success: 429 (Too Many Requests).`
**ProblemId:** `System.Net.Http.HttpResponseMessage.EnsureSuccessStatusCode:77`

### Probable causes
- The application is exceeding the rate limit of an external API (e.g. Azure OpenAI, a payment gateway, a third-party SaaS).
- Burst traffic or a background job is issuing too many requests in a short window without honouring `Retry-After` headers.
- Missing or misconfigured retry/throttling policy.

### Remediation ideas
- Identify which `HttpClient` named instance or base URL is producing 429s by checking Application Insights dependency telemetry or enriching the exception with the target URL.
- Implement a Polly `RateLimiter` or `WaitAndRetry` policy that reads the `Retry-After` response header.
- Add circuit-breaker logic so repeated 429s temporarily halt outbound calls and reduce the blast radius.
- If the target is Azure OpenAI, consider provisioned throughput (PTU) or request caching for repeated prompts.
